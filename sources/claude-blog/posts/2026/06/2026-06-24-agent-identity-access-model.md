# Claude Tag 中的智能体身份：面向自主、团队级 AI 的全新访问模型

> 来源：Claude Blog，2026-06-24
> 原文链接：https://claude.com/blog/agent-identity-access-model
> 分类：Claude Code

## 导语

Claude Tag 的智能体身份（agent identity）访问模型如何运作，以及在团队工作区中配置它的最佳实践。

## 核心要点

- 在多人协作（multiplayer）的 AI 体验中，"以用户身份行事"的传统模型行不通，因为智能体自主性不断提升，且共享频道中有多人同时操作，无法确定该套用谁的权限。
- 智能体身份模型让 Claude 拥有自己的账户：它以 Claude 应用身份在 Slack 发帖、以 Claude GitHub 应用身份开 PR、以管理员配置的服务账户查询数据仓库。
- 访问问题从"这个用户能做什么"转变为"这个智能体在这个隔间里能做什么"，权限从"按用户"转为"按频道"。
- 管理员在工作区层级定义身份基线，频道默认继承，并可在频道层级覆盖；可配置仓库访问、连接器（connectors）、技能与插件、常驻指令等。
- 私有频道各自拥有独立身份，公共频道共享工作区级身份；记忆与访问均遵守这些边界。
- 建议从宽松访问起步，再依据组织管理偏好收紧；每一次授权都谨慎、逐步进行，并配合审计追踪。
- 私信（DM）运行在用户个人 claude.ai 账户上，与共享频道机制不同；未来将引入即时凭证授权和身份感知叠加层。

## 中文译文

要让 AI 智能体在人机协作团队中发挥最佳表现，它需要拥有与人类相同的工具、文档和上下文访问权限。

在"单人"（single player）AI 体验中（一个人与一个助手对话），这很简单：你连接自己的账户，智能体便代表你行事。但在像 Claude Tag 这样的"多人"（multiplayer）AI 体验中，Claude 与许多人同时处于一个共享频道里，它依赖的是属于该工作区（workspace）的工具和上下文，而非任何个人的。

为了让多人体验得以运作，Claude 需要为这些工具拥有自己的账户，由管理员设置并绑定到工作区。我们称这种访问模型为智能体身份（agent identity）。

在本文中，我们将解释智能体身份如何运作、它如何把权限从"按用户"转为"按频道"，以及如何在你自己的工作区中恰当地划定其范围。

### 为何"以用户身份行事"会失效

当你把 AI 当作个人助手时，你可以连接 Google Drive、GitHub 和日历等平台，让模型使用你的访问权限去读写其中的内容。

这一模型在 Claude Tag 中行不通，原因有二：

- 不断提升的智能体自主性。AI 智能体能够独立可靠完成的任务时长，大约每四个月翻一番。智能体如今会为自己安排稍后执行的任务，并在提出请求的人早已下线之后响应事件。虽然用户会设置在特定情境下触发其行动的例程，但智能体在很大程度上是自主工作的。

- 多人团队。Claude Tag 把 Claude 置于团队已经在协作的共享空间中——例如一个三名工程师和一名产品经理一起调试的频道。但当不止一个人在指挥时，该套用谁的权限？没有哪个人选在所有时刻都正确。这让管理员能够独立于参与的人类去定义智能体在 Slack 中能做什么，并对在 Slack 中所做之事进行独立追踪。

#### Claude 以自身身份行事

在一个启用了 Claude Tag 的频道中，Claude 并非代表某一个用户行事。它在所接触的每个系统中都有自己的账户：它以 Claude 应用身份在 Slack 发帖，以 Claude GitHub 应用身份开拉取请求（pull request），并以管理员配置的服务账户查询你的数据仓库。

由于不涉及任何个人用户凭证，共享频道永远不会变成通往某人私人文档的侧门。

#### 继承权限

在智能体身份模型中，管理员在工作区层级定义一个身份——即 Claude 在各处持有的连接与技能的基线集合——而每个频道默认继承它。然后，在合理之处，他们可以在频道层级进行覆盖，例如授予工程频道访问 GitHub 和数据仓库的权限，或将某个 CRM 连接限定在单一私有频道内。

除凭证之外，管理员还可定义：

- 仓库访问：Claude 可以读写哪些代码仓库。

- 连接器（Connectors）：Claude 用于完成工作的工具和 API 密钥。在整个组织中，不同的 API 密钥可以以不同的权限级别连接到同一服务（例如，Claude 在通用频道中可能被赋予数据仓库的只读访问权限，而在数据团队的私有频道中被赋予写权限）。

- 技能与插件（Skills and plugins）：包含指令、脚本和资源的文件夹，Claude 会动态加载它们以提升在专业任务上的表现。

- 常驻指令（Standing instructions）：为每个频道定制的指令和上下文。

由于这一模型围绕彼此独立的 Claude 身份运作，撤销某个身份就会终止 Claude 在使用该身份的所有地方的访问权限。相比在数十个用户账户间审计单个智能体动作，这种管理方式所需的工作量要小得多。

### 智能体身份模型如何运作

智能体身份把"这个用户能做什么"这一问题替换为"这个智能体在这个隔间里能做什么"。这背离了按用户的访问控制列表（Access Control List）：它意味着，一个对仓库没有直接访问权限的频道成员，只要该频道的配置文件授予了 Claude 该权限，就可以请求 Claude 读取那个仓库。

这并不寻常，但我们认为这是迈向一种适用于自主、多人智能体的访问模型的必要一步。下面，我们勾勒出该如何思考这些边界的设定。

#### 身份边界如何运作

Claude Tag 为每个私有频道创建一个独立身份；工作区中的公共频道共享一个工作区级身份。Claude 在法务频道中的身份无法触及未在该处授予的代码，它在工程频道中的身份也无法读取未在该处授予的法务文档。记忆与访问都遵守这些边界：Claude 在私有频道中所学到的内容，绝不会出现在更广的工作区中。

身份归属于频道，因此默认情况下频道内任何人都可以 @ Claude，而管理员可以把每个频道的配置文件限定到权限最低的成员。在企业版（Enterprise）计划中，基于角色的访问控制（role-based access control）让管理员可以更进一步，决定哪些成员可以调用 Claude，从而让一个频道既管控智能体能触及什么，也管控谁能发起请求。

#### 对工具与上下文的宽松默认访问

![Image 1](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a3a0eae87418506ee6e5b08_agent-access-scoping%20(2).jpg)

在 Anthropic 内部运行 Claude Tag 时，我们发现其价值会随着工具和上下文访问的增加而叠加放大。每个接入的系统都让其他所有系统更有用，因为 Claude 可以跨系统组合上下文——把 Slack 中的一段对话、Drive 中的一份文档、追踪器中的一张工单以及数据仓库中的一次查询，整合成任何单一工具都无法提供的答案。

最能从 Claude 获益的团队，是那些从一开始就慷慨授予访问权限、再根据组织管理偏好收紧访问的团队。智能体身份赋予管理员足够宽泛的范围，让 Claude 能完成有用的跨系统工作，同时边界足够牢固，使访问绝不会流向未被授予之处。我们的建议是：先在几个频道中以一份基线配置文件起步，查看审计追踪记录，然后在工作需要之处扩展访问——每次都谨慎地逐项授权。

对于需要更细粒度的组织，管理员可以在特定频道中禁用 Claude Tag。管理员还可以应用基于角色的访问控制（RBAC），将 Claude Tag 的访问权限限定给特定用户。

#### 私信

在 Claude Tag 中，私信（direct message）的运作方式与共享频道不同。私信运行在用户各自的 claude.ai 账户上——使用他们的连接器、凭证，结果上也署他们的名字。这使得私信成为与 Claude 处理那些绝不应存在于频道中的任务和工具的合适场所，比如邮件草稿，或只有你自己持有许可证的软件。

#### 安全与审计

当管理员将某个连接添加到频道的配置文件时，该凭证会被独立存储并映射到该频道的身份，然后在请求时于网络边界处注入。所有通往管理员未允许的主机的出站流量都会被直接阻断。在审计方面，使用智能体凭证进行的每一次例程、记忆写入和网络调用都会被记录；并且由于 Claude 以自己的服务账户行事，这些动作也会落入各个接入系统自身的日志中。

### 接下来

智能体身份是 Claude Tag 访问模型的基础。未来，我们计划增强 Claude Tag 的安全功能，纳入即时凭证授权（just-in-time credential grants）——使用户能够在当下批准单次敏感操作，而无需永久性地扩大智能体的范围——以及面向具有更复杂权限结构的组织的身份感知叠加层（identity-aware overlay）。这将在智能体范围之上增加用户级别的检查，使 Claude 只有在频道配置文件和发起请求的用户自身权限都允许时才会行动。

像 Claude Tag 这类产品从单人 AI 向多人 AI 的转变，使长期运行、基于团队的工作成为可能。智能体身份确保 Claude 对工具的访问既足够宽泛而有用，又限定得足够好，从而在企业规模下保持安全。
