# Claude Tag 中的智能体身份：面向自主、团队级 AI 的全新访问模型

> 来源：Claude Blog，2026-06-24
> 原文链接：https://claude.com/blog/agent-identity-access-model
> 分类：Claude Code

## 核心要点

- 在人机协作团队中，AI 智能体（agent）需要与人类拥有相同的工具、文档和上下文访问权限，才能发挥最佳效果。
- 传统的"以用户身份代为操作"模型在多人共享（multiplayer）场景下失效，原因在于智能体自主性的提升以及多人共同操控时无法确定该套用谁的权限。
- 智能体身份（agent identity）模型让 Claude 以自身的账户在各系统中操作，由管理员在工作区（workspace）层级配置，并可在频道（channel）层级覆盖。
- 该模型将"这个用户能做什么"的问题替换为"这个智能体在这个隔离区里能做什么"，权限从按用户转为按频道划分。
- 私有频道拥有独立身份，公共频道共享工作区级身份；记忆与访问均遵守这些边界。
- 直接消息（DM）运行在用户各自的 claude.ai 账户上，使用其个人连接器与凭证。
- 凭证独立存储并在网络边界按请求注入，所有操作均被审计记录；未来计划引入即时（just-in-time）凭证授予与身份感知叠加层。

## 中文译文

要让一个 AI 智能体在人机协作团队中发挥最佳水平，它需要拥有与人类相同的工具、文档和上下文访问权限。

在"单人"AI 体验中（一个人与一个助手对话），这很简单：你连接自己的账户，智能体便代表你行动。但在像 Claude Tag 这样的"多人"AI 体验中，Claude 与许多人同处一个共享频道，它所依赖的工具和上下文属于整个工作区，而非任何单个个体。

为了让多人体验得以运作，Claude 需要拥有自己的工具账户，由管理员设置并绑定到工作区。我们把这种访问模型称为智能体身份（agent identity）。

在本文中，我们将解释智能体身份如何运作、它如何将权限从按用户划分转为按频道划分，以及如何在你自己的工作区中合理地界定其范围。

## 为什么"以用户身份操作"会失效

当你把 AI 当作个人助手使用时，你可以连接 Google Drive、GitHub 和日历等平台，让模型使用你的访问权限在其中读写。

这种模型在 Claude Tag 中行不通，原因有二：

- **智能体自主性的提升。** AI 智能体能够可靠独立完成的任务时长，大约每四个月翻一番。智能体现在会为稍后安排自己的任务，并在提出请求的人早已下线许久之后响应事件。虽然用户设置了在特定情境下触发其行动的例程（routine），但智能体在很大程度上是自主工作的。

- **多人团队。** Claude Tag 把 Claude 放进团队已在协作的共享空间——例如，一个由三名工程师和一名产品经理一起调试的频道。但当不止一个人在操控时，该套用谁的权限？没有任何单一的人选在所有时候都正确。这让管理员能够独立于参与的人类，定义智能体在 Slack 中能做什么，并对在 Slack 中所做之事进行单独的追踪。

### Claude 以自身身份行动

在一个启用了 Claude Tag 的频道中，Claude 并非代表某个单一用户行动。它在接触的每个系统中都拥有自己的账户：它以 Claude 应用的身份在 Slack 中发帖，以 Claude GitHub 应用的身份打开拉取请求（pull request），并以管理员配置的服务账户（service account）查询你的数据仓库（warehouse）。

而且，由于不涉及任何个人用户凭证，共享频道永远不可能变成通往某人私人文档的侧门。

### 继承权限

在智能体身份模型中，管理员在工作区层级定义一个身份——即 Claude 在各处持有的连接与技能的基线集合——每个频道默认继承它。然后，在合理之处，他们可以在频道层级覆盖它，例如授予工程频道访问 GitHub 和数据仓库的权限，或将某个 CRM 连接限定在单个私有频道内。

除了凭证之外，管理员还可定义：

- **仓库访问权限：** Claude 可以读写哪些代码仓库。

- **连接器（connector）：** Claude 用于完成工作的工具和 API 密钥。在整个组织内，不同的 API 密钥可以以不同的权限级别连接到同一个服务（例如，Claude 在通用频道中可能被授予只读的仓库访问权限，而在数据团队的私有频道中被授予写入权限）。

- **技能与插件（skill and plugin）：** Claude 动态加载的指令、脚本和资源文件夹，用以提升其在专门任务上的表现。

- **常驻指令（standing instruction）：** 针对每个频道的自定义指令与上下文。

由于该模型围绕着各自独立的 Claude 身份运作，撤销某个身份就会在该身份曾被使用的所有地方终止 Claude 的访问权限。相比于审计分布在数十个用户账户中的单个智能体操作，这种管理方式所需的工作量要小得多。

## 智能体身份模型如何运作

智能体身份将"这个用户能做什么？"的问题替换为"这个智能体在这个隔离区里能做什么？"。这背离了按用户划分的访问控制列表（Access Control List）：它意味着，一个对仓库没有直接访问权限的频道成员，可以请求 Claude 读取该仓库——只要该频道的配置文件授予了 Claude 这一权限。

这并不常见，但我们认为这是迈向一种适用于自主、多人智能体的访问模型的必要一步。下面，我们将勾勒出如何思考这些边界的设定。

### 身份边界如何运作

Claude Tag 为每个私有频道创建一个独立的身份；工作区中的公共频道共享一个工作区级的身份。Claude 在法务频道中的身份无法触及未在该处授予的代码，而它在工程频道中的身份也无法读取未在该处授予的法务文档。记忆与访问都遵守这些边界：Claude 在私有频道中学到的东西绝不会出现在更广的工作区中。

身份属于频道，所以默认情况下其中的任何人都可以标记（tag）Claude，而管理员可以将每个频道的配置文件限定到权限最低的成员。在企业版（Enterprise）计划中，基于角色的访问控制（role-based access control）让管理员能更进一步，决定哪些成员可以调用 Claude，从而让频道既管控智能体能触及什么，也管控谁可以发起请求。

### 对工具与上下文的宽泛默认访问

在 Anthropic 内部运行 Claude Tag 时，我们发现它的价值随着工具和上下文的访问而复合增长。每一个连接的系统都让其他系统更有用，因为 Claude 可以跨系统组合上下文——从 Slack 中提取一个讨论串、从 Drive 中提取一份文档、从跟踪器中提取一个工单、从仓库中提取一次查询，将它们汇成一个任何单一工具都无法提供的答案。

最能从 Claude 中获益的团队，是那些从一开始就慷慨授予它访问权限、然后根据组织管理员偏好逐步收窄访问的团队。智能体身份给予管理员足够宽泛的范围，让 Claude 能完成有用的跨系统工作，同时边界又足够牢固，使访问权限绝不会流向未被授予的地方。我们的建议是：先在几个频道中以一个基线配置文件起步，阅读审计轨迹，然后在工作有需要之处扩展访问——一次一个深思熟虑的授予。

对于需要更细粒度的组织，管理员可以在特定频道中禁用 Claude Tag。管理员还可以应用基于角色的访问控制（RBAC），将 Claude Tag 的访问限定给特定用户。

### 直接消息

在 Claude Tag 中，直接消息（direct message）的运作方式与共享频道不同。DM 运行在用户各自的 claude.ai 账户上——使用他们的连接器、凭证，结果上也署他们的名字。这使得 DM 成为与 Claude 处理那些绝不应存在于频道中的任务和工具的合适场所，比如电子邮件草稿，或只有你自己拥有许可证的软件。

### 安全与审计

当管理员将一个连接添加到某频道的配置文件时，该凭证会被独立存储并映射到该频道的身份，然后在请求时于网络边界处注入。流向任何管理员未允许的主机的出站流量会被直接阻断。在审计方面，使用智能体凭证发出的每一个例程、每一次记忆写入和每一次网络调用都会被记录，而且由于 Claude 以自己的服务账户行动，这些操作也会落入每个连接系统自身的日志中。

## 下一步

智能体身份是 Claude Tag 访问模型的基础。未来，我们计划强化 Claude Tag 的安全产品，使其包含即时（just-in-time）凭证授予——让用户可以在当下批准单次敏感操作，而无需永久扩大智能体的范围——以及面向具有更复杂权限结构的组织的身份感知叠加层。这将在智能体的范围之上增加用户级的检查，使 Claude 只在频道的配置文件和发起请求的用户自身权限同时允许时才行动。

从单人 AI 到多人 AI 的转变，在 Claude Tag 这样的产品中让长期运行、基于团队的工作成为可能。智能体身份确保 Claude 对工具的访问足够宽泛以发挥价值，又足够受限以在企业规模下保持安全。
