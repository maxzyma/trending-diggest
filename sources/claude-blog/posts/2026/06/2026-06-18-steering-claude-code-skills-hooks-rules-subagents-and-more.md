# 驾驭 Claude Code：技能、钩子、子智能体及其他七种指令方式

> 来源：Claude 官方博客（claude.com/blog），2026-06-18
> 原文链接：https://claude.com/blog/steering-claude-code-skills-hooks-rules-subagents-and-more
> 分类：Claude Code · 阅读时长约 5 分钟

## 核心观点

**Claude Code 提供七种方式来定制 Claude 的行为，它们的本质区别在于三个维度：指令何时加载进上下文、是否能在长会话压缩（compaction）后保留、以及携带多大的"指令权威"。**

1. **七种方式**：CLAUDE.md 文件、规则（Rules）、技能（Skills）、子智能体（Subagents）、钩子（Hooks）、输出样式（Output Styles）、追加系统提示词（Append System Prompt）。
2. **选址原则**：事实性、全程需要的信息放 CLAUDE.md；流程性的放技能；约束类放规则（用 `paths` 限定范围）；必须确定性执行/强制禁止的放钩子或权限。
3. **关键洞察**：「每次 X 都要做 Y」和「绝对不能做 X」不该写进 CLAUDE.md——前者该用钩子，后者该用确定性的钩子或权限护栏，因为提示词约束在长会话、模糊情境或提示注入下会失效。

## 速查表：七种方式的关键差异

| 方式 | 何时加载 | 压缩行为 | 上下文成本 | 适用场景 |
|------|---------|---------|-----------|---------|
| **CLAUDE.md（根目录）** | 会话开始；全程驻留 | 记忆化（memoized），读一次缓存整个会话；压缩后清缓存重读 | 高。每行都耗 token，无论是否相关 | 构建命令、目录结构、monorepo 布局、编码规范、团队约定 |
| **CLAUDE.md（子目录）** | 按需——当 Claude 读取该子目录下文件时 | 直到再次触及该子目录前丢失 | 低。仅在处理相关子目录时占用 | 子目录专属约定 |
| **规则 Rules** | 会话开始（用户级）或仅当触及匹配文件时（路径限定） | 压缩时重新注入 | 中。除非路径限定，否则常驻 | 特定约束或约定（如所有 API handler 必须用 Zod 校验输入） |
| **技能 Skills** | 名称和描述在会话开始加载；完整正文在调用时加载 | 已调用的技能在共享预算内重新注入，最旧的先丢弃 | 低。正文仅在调用时加载；受跨技能共享 token 预算约束 | 流程化工作流（部署、发布清单） |
| **子智能体 Subagents** | 名称、描述、工具列表在会话开始加载；正文仅在通过 Agent 工具调用时加载 | 仅最终消息（摘要 + 元数据）返回主会话 | 低。被调用前在主上下文中零成本；运行于独立上下文窗口 | 并行运行或需隔离、只返回摘要的支线任务（深度搜索、日志分析、依赖审计） |
| **钩子 Hooks** | 在生命周期事件触发 | 完全绕过压缩 | 低。配置存在于主上下文之外；部分输出可能返回（如阻断错误） | 确定性自动化：运行 linter、完成时发 Slack、阻断命令、PreCompact 时备份聊天记录 |
| **输出样式 Output Styles** | 会话开始；注入系统提示词 | 从不压缩 | 高。占用上下文窗口，但会覆盖默认系统提示词 | 重大角色切换（从代码助手变为通用助手） |
| **追加系统提示词** | 会话开始；通过 CLI 参数传入 | 从不压缩；仅作用于该次调用 | 中。首次请求后缓存 | 语气、回复长度、格式偏好 |

## 七种指令传递方式详解

### 1. CLAUDE.md 文件

CLAUDE.md 是项目根目录下的 markdown 文件。它在会话开始时加载进上下文，并全程驻留。构建命令、目录结构、monorepo 布局、编码规范、团队约定都天然适合放在这里。

有两种类型，加载方式不同：

- **始终加载**：第一种是根目录 CLAUDE.md 文件，可以在共享仓库里，也可以本地保存你针对某项目的个人偏好。这些文件都在会话开始时加载，且不会在长会话中丢失或退化。当 Claude Code 压缩对话时，会重新读取这些文件。
- **按需加载**：位于你初始化会话的目录下方的子目录中的 CLAUDE.md 文件。例如 `app/api/CLAUDE.md` 会在 Claude 读取 `app/api` 下文件时加载，而非会话开始时。它的压缩行为与路径限定规则相同：直到再次触及该子目录前都不存在。cwd 下方所有子目录的 CLAUDE.md 都会在 Claude 读取该目录内文件时加载。

在共享仓库里，CLAUDE.md 会像任何无人负责的配置文件一样膨胀：每个团队都追加自己的指令，没人删东西。规模化后成本会复合叠加。每一行都会加载进每位在该仓库工作的工程师的每个会话，无论是否与其任务相关。这会消耗 token，并稀释 Claude 对真正重要指令的遵循度。随着文件增大，应把团队专属约定下沉到路径限定规则、把流程下沉到技能中，让它们只在相关时加载。

**提示**：把 CLAUDE.md 控制在 200 行以内，给它指定一个负责人，像审查代码一样审查它的改动。把这个文件想象成给 Claude 提供代码库的概览，或一个指向其他文件的索引——Claude 需要时可以从那些文件找到更多信息。

在 monorepo 中，给每个团队的目录配各自的子目录 CLAUDE.md，这样团队只加载自己的约定；开发者还可以用 `claudeMdExcludes` 设置跳过那些自己从不接触的团队的文件。

对于必须应用到组织内每个仓库的标准——安全策略、合规要求——可以通过 MDM 或配置管理把一份集中管理的 CLAUDE.md 部署到开发者机器上，且无法被个人设置排除。

> 更多 CLAUDE.md 设置详见官方博客《CLAUDE.md files: Customizing Claude Code for your codebase》。

### 2. 规则 Rules

**规则**是 `.claude/rules/` 中的 markdown 文件，给 Claude 提供特定的约束或约定。

未限定范围的规则行为类似 CLAUDE.md：在会话开始时始终加载，压缩时重新注入。这可能在任务无关时也加载上下文，浪费 token。

**路径限定规则**让你只在相关时加载规则指令——通过添加 `paths` 字段控制何时加载。例如：一个限定到 `src/api/**` 的规则在纯文档会话中不进上下文，只在 Claude 读取 `src/api/` 目录内文件时才加载。

格式如下：

```markdown
---
paths:
  - "src/api/**"
  - "**/*.handler.ts"
---
所有 API handler 在处理前必须用 Zod 校验输入。
```

**提示**：像"迁移文件只能追加（append-only）"这样针对特定文件的约束，最适合作为**规则**放在 `paths:` frontmatter 里。当指令涉及横切关注点、或出现在代码库多个（但非全部）角落的文件时，优先选路径限定规则，而非嵌套的 CLAUDE.md 文件。

### 3. 技能 Skills

**技能**位于 `.claude/skills/`，是包含指令、脚本和资源的文件夹，Claude 会动态加载。每个技能有一个 `SKILL.md` 文件，含名称、描述和正文。

只有名称和描述在会话开始时加载；完整正文在 Claude 调用技能时才加载——可以通过斜杠命令（如 `/code-review`）调用，也可以通过任务自动匹配触发。

例如，`/code-review` 是一个内置技能，它审查当前 diff 并报告发现，但不修改文件。技能定义了"剧本"，因此每次调用 Claude 都遵循同样的结构化流程。

压缩时，Claude Code 会在所有已调用技能的总预算内重新注入它们。如果你在一个会话中调用了很多技能，最旧的会先被丢弃。

**提示**：流程化的指令——如部署流程、发布清单、审查流程——应放在技能里，而非 CLAUDE.md。Claude Code 自带技能，你也可以编写自定义技能。

### 4. 子智能体 Subagents

**子智能体**是 `.claude/agents/` 中的 markdown 文件，定义用于特定支线任务的隔离助手。每个文件用 YAML frontmatter（name、description，外加可选的 model 和工具访问字段），后接一段正文，这段正文成为该子智能体的系统提示词。

子智能体与技能相似——名称、描述、工具列表在会话开始时加载，但正文里更大的上下文不会自动调用。Claude 通过 Agent 工具调用它们，传入一个提示词字符串。

子智能体正文里更大的指令上下文不仅不会自动调用，它**根本不会进入父对话**。

子智能体随后在自己全新的上下文窗口中运行，唯一返回主会话的，是子智能体的最终消息（通常是许多子任务的聚合结果）加上元数据。

这种模式可扩展：子智能体最多可嵌套五层，动态工作流可编排数十到数百个后台智能体，而无需你指定子智能体架构的每个细节。编排计划和中间结果存在于脚本变量中，而非 Claude 的上下文窗口里——这使得在不损失指令保真度的前提下实现规模化。

**提示**：这种隔离正是选用子智能体而非技能的主要原因之一。当深度搜索、日志分析、依赖审计等支线任务会用你不再引用的中间结果塞满主对话时，用子智能体。当你希望流程在主线程中展开、以便你能看到并引导每一步时，用技能。

### 5. 钩子 Hooks

**钩子**是用户定义的命令、HTTP 端点或 LLM 提示词，通过在 Claude 生命周期的特定事件（如文件编辑、工具调用、会话开始）触发，对 Claude 的行为提供更确定性的控制。

你可以在 `settings.json`、托管策略设置（managed policy settings）、或技能/智能体的 frontmatter 中注册钩子。

钩子有几种类型：command、HTTP、mcp_tool、prompt、agent。所有钩子都是确定性触发的。前三种确定性执行；后两种（prompt 和 agent）用 Claude 的判断而非一套规则来决定输出。

钩子上下文成本低，因为配置或指令存在于主上下文窗口之外。harness 运行处理器（command、http、mcp_tool），或用独立窗口发起模型调用（prompt、agent），取决于钩子类型。

部分钩子的输出可能会被保存到主上下文窗口。例如，一个阻断型钩子的标准错误会被保存进上下文，好让 Claude 知道调用为何被拒。但大多数钩子不会把输出保存进主窗口，除非配置显式返回。比如你用 `PreCompact` 事件在压缩前把聊天记录备份到另一个文件，Claude 并不会知道哪个文件存了聊天记录。

这一点使这些钩子类型与 CLAUDE.md、规则、技能有本质区别。

**提示**：把任何应当确定性发生的事交给钩子：编辑后运行 linter、完成时发 Slack、或在特定命令执行前阻断它。一个 `PreToolUse` 钩子可以检查任意工具调用，并通过退出码 2 拒绝它。钩子上下文成本低，因为它们是 harness 运行的代码，而非加载进上下文的、给 Claude 的指令。

### 6. 输出样式 Output Styles

**输出样式**是 `.claude/output-styles/` 中的文件，会把指令注入系统提示词。它们从不被压缩，在每个会话开始时加载，首次请求后缓存——因此上下文成本为中等。

因为它们位于系统提示词中，输出样式在我们目前介绍的所有方式中携带最高的指令遵循权重，应审慎使用。

**对输出样式的改动会替换默认输出样式**（除非你在样式 frontmatter 中设置 `keep-coding-instructions: true`）。在 Claude Code 中，这会移除那些告诉 Claude "它在帮用户做软件工程任务"的指令，以及其他关键默认指令，例如：

- 如何界定改动范围；
- 何时添加或省略代码注释；
- 如何处理安全问题；以及
- 验证习惯，如在宣告工作完成前先运行测试。

默认情况下，自定义输出样式会丢弃所有这些，Claude Code 会从软件工程师助手更多地变成通用助手。

**提示**：在编写自定义输出样式前，先看看内置样式。**Proactive**（主动）、**Explanatory**（讲解）、**Learning**（学习）涵盖了最常见的需求（自主性、教学模式、协作编码），你无需维护一个样式文件。

### 7. 追加系统提示词 Append System Prompt

修改输出样式之外的另一个选择是 `append-system-prompt` 参数。修改输出样式文件可能对 Claude 行为产生巨大的、意料之外的改变，而 append 参数只对原系统提示词做加法。它不修改 Claude 的角色，只在其默认角色上追加指令。

它也是在调用时传入，仅作用于该次调用，而非作为文件跨会话持久化。

追加系统提示词相比其他传递指令的方式可能有更高的上下文成本。它会增加输入 token，不过首次请求后提示缓存会降低这个成本。指示 Claude 使用更冗长或更长的风格也会增加输出 token。

**提示**：追加系统提示词最适合添加特定编码标准、输出格式或领域知识。请记住，追加系统提示词对遵循度有边际递减效应。一般而言，你用这种方式提供的指令越多，Claude 遵循得越不严格——尤其当有任何指令相互矛盾时。

## Claude Code 定制的快速建议

如果你发现自己在做以下事情之一，或许该考虑把指令放到别处：

**在 CLAUDE.md 里写"每次 X 都要做 Y"**。如果某个行为应当可靠地发生——比如每次编辑后运行 prettier、或完成时发 Slack——请改用 `settings.json` 里的钩子。模型"选择"去运行格式化工具，和格式化工具"自动"运行，是两回事。

**在 CLAUDE.md 里写"绝对不要做这个"**。当有某件事绝对不能发生时，指令是错误的工具。Claude 大多数时候会遵循指令，但在压力下、长会话中、模糊情境里，或因任务中访问的文件里存在提示注入（prompt injection）时，模型可能不遵循一条被提示的规则。真正的护栏必须是确定性的，而强制执行的方式是钩子和权限。一个 `PreToolUse` 钩子可以检查调用并以退出码 2 阻断它。**托管设置（Managed settings）**更进一步：它们由管理员部署、无法被用户本地配置覆盖，是唯一能强制实现确定性、组织级护栏的方式。

**在 CLAUDE.md 里写一个 30 行的流程**。流程属于技能。CLAUDE.md 是为 Claude 应当时刻持有的事实准备的：构建命令、monorepo 布局、团队约定。部署手册或安全审查清单应当放在 `.claude/skills/` 里，那里正文只在调用时加载。

**一条针对特定 API 却没有路径的规则**。如果一条规则只适用于 `src/api/**`，用 `paths:` 限定它的范围能让它在无关工作时不进上下文。一条未限定范围的规则，在机制上等同于把内容放进 CLAUDE.md：始终加载，始终耗 token。

**把个人偏好写进项目级 CLAUDE.md 文件**。所有基于文件的方式都有一个用户级对应版本，会为每个 Claude Code 会话加载，无论你在哪个仓库。把个人偏好（如始终使用语义化的 commit message）放在本地文件里。项目级文件留给团队范围内、但特定于某代码库的偏好。

## 开始上手

你可以在官方《Claude Code 最佳实践》文档中找到更多让 Claude Code 物尽其用的技巧和模式，从配置环境到跨并行会话扩展。

一旦你跑通了其中几样，就可以把它们中的许多（技能、子智能体、钩子、输出样式）打包成一个**插件（plugin）**，从而把一套连贯的配置分享给队友或项目。
