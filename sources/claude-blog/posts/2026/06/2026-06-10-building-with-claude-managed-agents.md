# 智能体形态的演进：基于 Claude 托管智能体（Claude Managed Agents）构建

> 来源：Claude Blog，2026-06-10
> 原文链接：https://claude.com/blog/building-with-claude-managed-agents
> 分类：Agents

## 核心要点

- 将智能体推向生产环境，所需的远不止一段好的提示词，还需要运行代码的环境、访问数据的凭据、可观测的会话以及随用量扩展的基础设施。
- Anthropic 的智能体构件经历了从简单 API、到 Claude Code、再到 Claude Agent SDK 的演进，逐步把经过调优的智能体框架（harness）交到开发者手中。
- 托管智能体的核心思路是「将大脑与双手解耦」：调用 Claude 的框架与执行代码的沙箱分离运行，二者通过仅追加的会话日志连接。
- 它围绕三种核心资源构建：智能体（agent）、环境（environment）和会话（session）。
- 相比自建框架，托管智能体在凭据隔离、降低延迟、可靠持久会话、以及托管/自托管部署灵活性上具有优势。
- Notion、Rakuten、Sentry、Asana、Atlassian 等客户已在生产中使用托管智能体交付产品。

## 中文译文

将智能体推向生产环境，所需的远不止一段好的提示词。智能体需要一个运行其所写代码的地方、访问你数据的凭据、可观测的会话，以及能随用量扩展的基础设施。在应用人工智能（Applied AI）团队，我们工作在产品、研究与基于 Claude 构建的客户三者的交汇处——我们反复看到同一个模式：基础设施才是区分原型与生产级智能体的关键。团队往往把开发周期消耗在安全、状态管理、权限控制和框架（harness）调优上。

**Claude 托管智能体（Claude Managed Agents）**是我们用于构建和部署生产级智能体的一套可组合 API。它将一个为性能调优的智能体框架与生产基础设施配对，使团队能够在数天而非数月内从原型走向上线。本文将介绍 Anthropic 智能体构件的演进、我们为何构建托管智能体，以及当下团队如何在生产中使用它。

### 演进智能体架构

2023 年我们向开发者开放 Claude 时，API 刻意保持简单：输入词元（tokens），输出词元。你发送一段提示，Claude 返回一段补全，框架和底层基础设施由你来构建。

多年来 API 稳步丰富，但底层契约从未改变：一次请求、一次模型回合，你的应用决定接下来发生什么。在很长一段时间里，这已经足够。总结一篇文档、对一张支持工单分类、改写一段文本——这类工作都能舒适地装进单个回合中。

然而随着时间推移，人们想要交付的任务不再能装进单回合。他们希望 Claude 把一项任务从头做到尾：查找某些信息、据此采取行动、观察发生了什么变化，再决定下一步做什么。而且他们希望它运行在工作本就所在的系统里，比如代码库、内部 wiki 或工单系统。

用 API 把 Claude 变成智能体，意味着要自己构建循环：询问模型该做什么、运行工具、把结果反馈回去，如此重复。你要负责构建和部署智能体脚手架，而随着模型演进，这套脚手架可能需要调优。对于需要完全定制的智能体，这种方式是合理的。但对于更可预测、更不复杂的智能体工作负载，随着模型和产品演进而不断优化框架就变得繁琐。

**Claude Code** 是我们在 2025 年推出的智能体编码工具，让 Claude 直接与你的代码库交互。它内含我们自己版本的框架：循环、工具执行、子智能体（subagents）、上下文管理，以及使其成为高效智能体的丰富能力。开发者自然希望在各自的领域中为自己的智能体获得类似的框架机制。

为了让团队能在 Claude Code 框架之上构建智能体，我们发布了 **Claude Agent SDK**。Claude Agent SDK 让开发者使用运行 Claude Code 的同一套机制来构建自己的智能体，而无需维护自制的循环。对许多团队来说，智能体正是从这一刻起变得实用：框架到手时已为 Claude 调优好，带有基础设施原语，并且会随 Claude Code 一同持续改进。

然而，即便有了框架，在生产环境中部署智能体仍可能面临诸多挑战：

- **托管与扩展（Hosting and scaling）。** 智能体在哪里运行，一个进程能为一项多小时的任务存活多久，用量增长时由什么来扩展它？
- **会话管理（Session management）。** 智能体的历史和进度存放在哪里？一次运行能否在中断后存活并无障碍地恢复？你能否回头检视先前会话中发生了什么？
- **文件系统管理（Filesystem management）。** 真正的工作意味着产出制品：编辑代码、写入文件、构建输出。智能体从哪里获得一个可操作的工作区，运行之间这个工作区又会怎样？
- **执行隔离（Execution isolation）。** Claude 写出的代码必须在某处执行。如果代码出错，影响范围（blast radius）有多大，在生产中你真正信任的边界又在哪里？
- **凭据（Credentials）。** 智能体需要访问你的系统。它如何在不向其生成的代码暴露专有信息的情况下获得访问权？
- **可观测性（Observability）。** 当一个智能体自主工作一小时并做出某些意外之举时，你能否重建它走过的每一步？

借助 Agent SDK，上述生产基础设施的许多要素通过 Claude Code 的机制提供。智能体获得一个真实的文件系统来工作，会话状态被持久化到本地或外部存储，可观测性可通过 OpenTelemetry 导出到你已在运行的任意监控栈中。

然而，随着团队越来越多地把智能体从本地开发推向生产，他们需要一种以托管基础设施大规模部署的方式。而随着模型及其周边框架日益先进——运行更久、执行更多代码、触及更多系统、采取更多行动——扩展、安全和沙箱化也变得更具挑战性。

其中一些障碍源自一个共同的架构选择：智能体框架往往与它所操作的文件系统运行在同一个容器内。容器必须先启动（付出启动成本）Claude 才能开始思考，智能体连同代码执行就紧挨着你的凭据，而当容器死亡时，运行也随之终止。

托管智能体通过**将大脑与双手解耦**来解决这些问题。调用 Claude 的框架与执行代码的沙箱分开运行，而会话——一份记录每次模型调用、工具调用和结果的仅追加日志——把二者连接起来。Claude 可以在任何容器存在之前就开始推理，沙箱远离你的凭据，整次运行可以在任意时刻从其会话中重建。

### 何时以及为何使用 Claude 托管智能体

在使用托管智能体构建时，用户定义任务、工具和护栏（guardrails），由 Anthropic 在我们的基础设施上运行智能体，并处理底层的智能体循环：如何给智能体提供调用工具的执行环境、如何在出错时恢复、多智能体编排（multi-agent orchestration）等等。

**当框架不随模型智能一同演进时，智能体就会失灵。** 在 Claude Sonnet 4.5 上，智能体在接近其上下文末尾时会急于完成，草草收尾而不是利用剩余的空间——这一模式被称为「上下文焦虑」（context anxiety）。我们的修复方法是在框架中加入上下文重置（context resets），内置了一个假设：Claude 在接近极限时需要帮助以保持连贯。这个假设没能挺过下一个模型。在 Claude Opus 4.5 上，该行为消失了，我们加入的重置只是额外开销。

对大多数组织而言，维护框架是一种并不能让其产品产生差异化的开销。框架必须针对特定模型行为调优；像压缩（compaction）、工具执行和缓存这类原语在 Claude 上的运作方式与其他模型不同。借助 Claude 托管智能体，框架随模型一同演进，使团队能够专注于真正让其智能体产生差异化的方面：上下文管理和领域专长。

为了让开发者能配置构建高效智能体所需的上下文和工具，托管智能体围绕三种主要资源构建：智能体、环境和会话。**智能体（agent）**是一份配置：一个模型、一段提示、一组工具，以及围绕它们的护栏。**环境（environment）**是智能体运行其中的执行上下文：沙箱容器、它的网络规则，以及其中预装的软件包，托管在我们的云上或你控制的基础设施上。每一次运行都是一个**会话（session）**，它将一个智能体与一个环境配对，并获得自己独立隔离的沙箱实例。会话在服务端持久化其完整事件历史、沙箱状态和输出，因此长时间运行的工作可以暂停、干净地恢复，并在事后逐步追溯。借助托管智能体，你可以一次性定义一个智能体和一个环境，然后随着工作负载增长，针对同一份配置运行许多会话。

### 在托管智能体上面向生产与规模构建

在应用人工智能团队内部，我们看到智能体从原型走向生产，既发生在 Anthropic 内部，也发生在客户的系统中，覆盖编码、金融、支持、法律以及十多个其他领域。这让我们清晰地看到，是什么把一个演示与一个生产就绪的智能体区分开来，以及团队常在哪里卡住。

下面我们分享在 Claude 托管智能体这类托管服务上构建的最常见原因：

1. **凭据被隔离在沙箱之外。** 当一切都运行在同一个容器中时，Claude 生成的代码就紧挨着你的凭据，因此提示注入（prompt injection）可能通过说服模型读取自身环境而导致它泄露某个令牌。我们可以通过在同一容器内设置稳健的护栏来防范，但解耦架构通过把凭据完全隔离在沙箱之外，实现了一种安全得多的方式。用于 MCP、CLI、GitHub 仓库等工具的令牌存放在一个独立的保险库（vault）中，由代理仅在需要时获取并解密。托管智能体提供开箱即用处理凭据的**保险库（Vaults）**，因此你无需自行运行密钥存储、无需在每次调用时传输令牌，也不会弄不清楚智能体是代表哪个终端用户行动的。保险库凭据在存储前以信封加密（envelope encryption）保护，检索需要一个用于验证的签名请求令牌。

2. **消除沙箱开销带来的更低延迟。** 延迟是许多企业团队最为关注的指标，因为用户在等待 Claude 响应时会有切身感受。没有托管智能体架构时，每个会话都得启动一个容器，即便是那些智能体只需思考、从不运行工具的会话。那段启动时间被白白浪费，用户会感受到首次响应前的延迟。借助托管智能体，Claude 立即开始推理，与此同时环境并行启动，而从不运行工具的会话则完全跳过容器。这意味着用户无需等待容器启动就能看到首个词元，而当智能体需要运行某些东西时环境也已就绪。在我们的测试中，这在中位情况（p50）下将首词元时间减少了约 60%，在最慢情况（p95）下减少了超过 90%。

3. **可靠、持久的会话，支撑会话管理、可观测性与记忆。** 托管智能体不以请求/响应方式思考，而是以事件（events）方式思考。一个会话是一条持续的事件流：每次模型调用、工具调用和结果，都被追加到一份存活于运行智能体的进程之外的日志中。借助这一架构，你能在智能体工作时随事件流入而获得实时更新，也能在之后恢复任意会话，无需管理任何数据库或存档点（save-points）。除非你删除会话，否则历史会在交互之间得以保留；当一个会话进入空闲，其容器会被建立检查点（checkpointed），以便你能从暂停处干净地继续。而且因为整次运行本身就是一份事件记录，可观测性与记忆随之而来：Claude 开发者控制台（Claude Developer Console）提供智能体会话的原生可视化时间线视图，以及一种可深入检视任意会话记录的调试体验。托管智能体还带有同样利用这种会话持久性的**记忆（Memory）**和**梦境（Dreaming）**等功能。梦境是一个定期进行的过程，它审阅你的智能体会话和记忆存储、提取模式并整理记忆，使你的智能体随时间改进。梦境在会话之间精炼记忆，从而通过读取持久会话日志，从反复出现的错误和用户偏好中得到改进。

4. **在 Anthropic 托管或自托管云容器之间的灵活性。** 默认情况下，借助托管智能体，你可以把编排和工具执行都委托给 Anthropic 托管的云容器。这让托管与扩展简单易行，提供一条更快通往生产的路径。因为在托管智能体中大脑与双手解耦，双手可以存在于任何地方，包括你的虚拟私有云（VPC）内部。因此，对于希望掌控工具执行的团队，我们也提供自托管沙箱（self-hosted sandboxes），使智能体的代码、文件系统和网络出口流量（network egress）永不离开其环境。我们还提供 **MCP 隧道（MCP tunnels）**，让你能把 Claude 连接到运行在你私有网络内部的模型上下文协议（Model Context Protocol，MCP）服务器。因此，自托管沙箱控制**智能体的代码在何处执行**，MCP 隧道控制 **Anthropic 如何触达你网络中的 MCP 服务器**，让你能够精确控制哪些东西留在你的边界之内。

除这些功能之外，更多能力还包括：让智能体根据评分标准（rubric）为自己的工作打分的「成果（outcomes）」、多智能体编排、权限策略，以及 webhooks。点此了解更多。

#### 客户当下如何在托管智能体上构建

跨各行各业，客户已经在用 Claude 托管智能体将智能体推向生产。以下是几个例子：

- **Notion** 在托管智能体上运行其自定义智能体（Custom Agents）：团队直接从任务看板把工作分配给 Claude，Claude 拾取围绕每项任务的文档、会议记录和已连接的数据，完成后的代码、演示文稿和网站回到工作区供审阅。数十个任务并行运行，他们的团队描述了一个早期原型把大约十二小时的工作变成二十分钟。
- **Rakuten** 使用托管智能体在产品、销售、营销和金融领域交付了专门的智能体，每一个都在约一周内上线。
- **Sentry** 把其 Seer 调试智能体与一个负责编写补丁并发起 PR 的 Claude 智能体配对，由一名工程师在数周而非数月内构建完成。
- **Asana** 构建了能在项目内部拾取任务的 AI Teammates，**Atlassian** 则把开发者智能体放入了 Jira 工作流。

## 开始使用 Claude 托管智能体

我们构建托管智能体，是为了让你尽可能轻松地通过 Claude Code 和 platform.claude.com 上的 Claude 开发者控制台启动智能体。例如，控制台的快速入门让你可以从一个智能体模板开始，或用自然语言描述一个智能体，然后在几分钟内把它变成一个你可以保护并部署的生产就绪智能体。

在 Claude Code 中，默认提供 `/claude-api` 技能（skill），它为 Claude 提供用于在 Claude 托管智能体上构建应用的详尽、最新的参考资料。我们强烈建议你利用它来获取设置托管智能体应用的最佳实践。运行 `/claude-api managed-agents-onboard` 即可开始一次以访谈驱动的演练，从零开始设置一个新的托管智能体。

## 构建托管智能体的未来

随着团队分享他们用托管智能体构建的东西，我们看到他们过去花在生产基础设施上的时间，如今转向了真正让其智能体产生差异化的方面：管理上下文和为用户量身定制体验。现在，当一个新模型发布时，你更新智能体以使用它、重新运行你的评估（evals），然后无需触碰底层架构就交付改进。
